# Past Modeling Experiments

The Data Department has implemented and tested numerous improvements to the model-res-avm repository across modeling, infrastructure, reporting, and feature engineering. Not all tests and experiments have been incorporated into the final model. This document serves as a historical record of what has been explored, what was adopted, and what was set aside.

| Question / Goal | Issue | Findings / Output | Theme | Date |
| --- | --- | --- | --- | --- |
| Should `bonsai` (the official Tidymodels/parsnip shim for LightGBM) replace `Lightsnip`? | [#16](https://github.com/ccao-data/model-res-avm/issues/16) | `{bonsai}` is promising but not ready for adoption — it lacks advanced LightGBM hyperparameters and early stopping. Sticking with `Lightsnip`. | <span style="color:#4A90D9">Modeling</span> | 12-05-2023 |
| Test a City of Chicago–only model (using expanded city-specific data and only City triad sales) and compare performance to city-only ratios from county-wide models. | [#28](https://github.com/ccao-data/model-res-avm/issues/28) | Tested two configurations (Chicago-only sales; Chicago-only sales + large cross-validation). No change in performance in either case. | <span style="color:#4A90D9">Modeling</span> | 01-18-2024 |
| Prior CCAO residential models have used a single model that encompasses both single- and multi-family properties. We may achieve better accuracy for multi-family properties if we instead separate them out in a separate model. | [#29](https://github.com/ccao-data/model-res-avm/issues/29) | The result is essentially nil — splitting the models has no significant impact on overall performance statistics. This is somewhat expected, as the model already has access to plenty of information to differentiate between single- and multi-family properties. | <span style="color:#4A90D9">Modeling</span> | 12-05-2023 |
| The `grid` argument of `tune_bayes()` can use a result from `tune_grid()`. We should use this to "seed" `tune_bayes()` with a manual or space-filling grid. | [#36](https://github.com/ccao-data/model-res-avm/issues/36) | The technique ended up being far slower than Bayesian optimization with early stopping. | <span style="color:#4A90D9">Modeling</span> | 12-25-2023 |
| Per recommendation from folks at posit::conf, test out the `Cubist` engine from the `rules` R package to see how it performs on our data. | [#37](https://github.com/ccao-data/model-res-avm/issues/37) | `Cubist` outperforms a linear model but not the current main model. Major drawbacks: single-threaded (very slow to train) and high memory usage (~250 GB), making a full grid search infeasible. | <span style="color:#4A90D9">Modeling</span> | 12-25-2023 |
| We may be able to cross-validate significantly faster using racing-based `tune` methods, which are available in the `finetune` R package. Test cross-validation techniques: standard grid search (with sub-model trick), tune racing, and simulated annealing. | [#38](https://github.com/ccao-data/model-res-avm/issues/38) | All techniques were slower than Bayesian optimization with early stopping. Sub-model fitting works but requires fitting on the maximum number of iterations. Grid racing is faster than standard grid search but has the same limitation. Simulated annealing is too slow. Bayesian optimization remains the best approach. | <span style="color:#4A90D9">Modeling</span> | 12-25-2023 |
| Revisit ensemble/stacked modeling using the `stacks` package (prior attempt with Tidymodels was inconclusive due to immaturity of the tooling at the time). | [#46](https://github.com/ccao-data/model-res-avm/issues/46) | The added complexity of a stacked model has not been worth it compared to a standard boosted tree model in prior testing. | <span style="color:#4A90D9">Modeling</span> | 12-06-2024 |
| Test categorical embeddings (via `embed.tidymodels.org`) as an alternative to LightGBM's native categorical handling. | [#49](https://github.com/ccao-data/model-res-avm/issues/49) | Embeddings performed worse than LightGBM's native handling. The `embed` package also has heavy dependencies. Embeddings removed entirely (including from the linear model). | <span style="color:#4A90D9">Modeling</span> | 01-27-2024 |
| Test a homeowner exemption model feature as a proxy for owner-occupied status. | [#50](https://github.com/ccao-data/model-res-avm/issues/50) | Feature added [here](https://github.com/ccao-data/data-architecture/pull/296/changes). | <span style="color:#8E44AD">Model Features</span> | 01-20-2024 |
| Test `Tidymodels` "1 standard deviation" rule selector to see if it improves generalization to a held-out validation set. | [#134](https://github.com/ccao-data/model-res-avm/issues/134) | The "best" model consistently generalizes better than the 1 SD selection. The 1 SD selector tends to prefer many shallow trees (3,000 iterations, <100 `num_leaves`) over fewer but deeper trees (1,500 iterations, <250 `num_leaves`). | <span style="color:#4A90D9">Modeling</span> | 02-07-2024 |
| Add a price-strata or lagged-price feature to give the model a starting price point, enabling it to segment the market by value tier. | [#160](https://github.com/ccao-data/model-res-avm/issues/160) | Two problems identified: sold and unsold properties are treated differently due to the 3-year assessment cycle, and the lag price feature effectively leaks the outcome into the assessment set as a predictor. Feature scrapped. | <span style="color:#4A90D9">Modeling</span> | 02-03-2024 |
| Update cross-validation: revert to rolling-origin splitting, make it non-cumulative, and store per-fold predictions and residuals. | [#161](https://github.com/ccao-data/model-res-avm/issues/161) | New sliding-window CV function added with: switchable random (v-fold) vs. recency-based (sliding window) splitting via `validation_type`; non-cumulative folds; and overlapping fold support. | <span style="color:#4A90D9">Modeling</span> | 01-16-2024 |
| Replace LightGBM's native missingness handling with more targeted imputation using `recipes::step_impute_*` functions. | [#162](https://github.com/ccao-data/model-res-avm/issues/162) | No meaningful performance difference. Missingness is limited and LightGBM handles it well natively. Advanced strategies (bagging, KNN) were too slow to test even on large AWS instances. | <span style="color:#4A90D9">Modeling</span> | 01-29-2024 |
| Test log-transforming the outcome (sale price) and skewed input features (income, building sqft, etc.) to improve model performance, particularly at the low end. | [#163](https://github.com/ccao-data/model-res-avm/issues/163) | Results are inconclusive. Slightly improves COD and low-end accuracy but worsens overall RMSE and high-end accuracy. See commit `8a7ae37` and runs `2024-01-24-sagacious-liz` / `2023-12-26-kind-caroline`. | <span style="color:#4A90D9">Modeling</span> | 01-23-2024 |
| Test smaller geographic units as location features: census tracts + PUMA, census block groups + PUMA, and Sidwell ID broken out by block. | [#166](https://github.com/ccao-data/model-res-avm/issues/166) | No meaningful improvements across all tested combinations. Census tracts as numeric yielded a minor improvement (<1%). Sidwell ID by block performed worse than standard location predictors. | <span style="color:#8E44AD">Model Features</span> | 01-17-2024 |
| The comps pipeline produces unexpected results for high-value cards, likely because tree weights are computed globally (based on training-set RMSE) rather than per observation. | [#193](https://github.com/ccao-data/model-res-avm/issues/193) | Updated the comps pipeline (`python/comps.py`) to use per-observation tree weights instead of global weights. | <span style="color:#E74C3C">Comps</span> | 02-03-2024 |
| Explore strategies for modeling multi-card properties, which are excluded from training because sale price cannot be attributed per card. | [#228](https://github.com/ccao-data/model-res-avm/issues/228) | Two best strategies identified: `single card summing sqft` and `multi-card sqft aggregation training`. The aggregation training method gives the best performance for multi-card properties but slightly degrades single-card prediction quality. | <span style="color:#4A90D9">Modeling</span> | 01-28-2025 |
| Test the effect of adding a random noise vector on feature importance and prediction stability. | [#272](https://github.com/ccao-data/model-res-avm/issues/272) | Adding random noise did not affect global performance stats but did expose prediction instability. A small number of predictions flipped between ~$100K and ~$10M, tracing back to erroneous high-value sales that had passed sales validation. Correcting the underlying sales stabilized predictions. | <span style="color:#4A90D9">Modeling</span> | 01-22-2025 |
